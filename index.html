watch one path propagate

A live blotter, ticking. Each update mutates exactly one path — trades[i].field = value — and only the affected cell flashes. Sibling fields are never visited; work is proportional to the path that changed, not the row.

idtenorbidasklastpnl

why incremental?

Work is proportional to the path that changed — not the row, not the dataset, not anything broader. When you mutate a deeply-nested property: 

trades[1234].bid = 99.85

…the underlying notification carries the exact path ['1234', 'bid'] and the new value. Each layer in the pipeline does work scoped only to that path: filter reruns its predicate for one row, between does a single bisect step, intersect flips one bitmask entry per source, za repositions one entry in its sorted index, and the DOM updates the single binding tied to the changed path. No diff pass, no list re-render, no key reconciliation.

A typical Redux + virtual-DOM stack re-runs the entire selector chain over all 5,000 trades on every tick. With one tick per millisecond, that doesn't scale at all. Operators here are written for minimum-work propagation by construction.

operators

Every operator below is a view derived from the same trades proxy backing the blotter above. Streaming bid/ask/pnl mutations propagate through each chain incrementally — no recomputation outside the changed path.

filter — pick a tenor

Rows matching a predicate. Click a chip to switch.


      

      

    


    

      
between — brush by pnl

      
Rows where a column falls in a range. Each dot is one
        trade at its current pnl; the orange band is the active range.

      

      

      

        
        
      

    


    

      
za — top by pnl

      
Sort descending, optionally limited. Top 5 winners,
        re-ranking live as pnl ticks.

      
trades.za('pnl', 5)

      

    


    

      
length — count, total or grouped

      
Total row count, or counts under a key.

      
trades.length()
trades.length(d => d.tenor)

      

    


    

      
intersect — filter ∩ filter

      
Rows present in every source view. Profitable trades at
        the chosen tenor.

      

      

      

    


    

      
group — nest by tenor

      
Rows nested under a computed key. Each tenor becomes its
        own bucket, with its trades indented underneath.

      
trades.group(d => d.tenor)

      

    


    

      
map — derive bid-ask spread

      
Per-row transform. The new spread
        column is computed from each row's bid & ask; it updates whenever
        either field ticks.

      
trades.map(d => ({ ...d, spread: d.ask - d.bid }))

      

    


    

      
to — aggregate pnl

      
Whole-value transform. One number derived from all rows;
        recomputes on every tick.

      
trades.to(arr => arr.reduce((s, t) => s + t.pnl, 0))

a reactive scalar


import { $, value } from 'data'

const count = $(0)
const doubled = count.to(n => n * 2)

const events = doubled.connect([])

count[value] = 5
count[value] = 7

// events:
// [ { type: 'update', key: [], value: 0  },
//   { type: 'update', key: [], value: 10 },
//   { type: 'update', key: [], value: 14 } ]


    
rendering to the dom


import { $, render, HTML } from 'data'
const { ul, li } = HTML

const todos = $([{ task: 'foo' }, { task: 'bar' }])

render(document.body,
  ul(todos, (node, item) => node.text(item.task))
)

todos.insert({ task: 'baz' })   // a new <li>baz</li> appears
